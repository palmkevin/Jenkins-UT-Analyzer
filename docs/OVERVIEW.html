Concept Overview · Architecture & Workflows

Jenkins UT Analyzer

Automates the daily triage of integration-test (“unittest”/UT) failures from a Jenkins pipeline — gathering the evidence, correlating causes, and learning from the human’s explanations — so the work that one person did full-time becomes fast, persistent and reproducible.

Why The parts System map Ingest loop Analysis Human triage Learning loop Alerting Reference Configuration Deployment

Why this tool exists

Integration tests run in a Jenkins job triggered by every commit to the SVN trunk. Before each build, a working ut database is synced from a reference database, Oracle ut_ref. Roughly 25 000 tests run in each of two parallel tracks (permanent and permanent_py39 — distinguished by execution environment, not by which tests they run) and merged into one report.

Historically one person monitored this full-time: after each build they compared failures to the previous build, isolated the new failures, and attributed each to a probable cause — a code change, a data change in ut_ref, infrastructure, or other — then contacted the responsible developer and watched the next build to confirm a fix. That triage was slow, person-dependent, and not reproducible.

The Analyzer automates the evidence-gathering and presents a clear, persistent picture per test and per build — and learns from the human’s confirmed explanations over time. It is deliberately assistive: it gathers, correlates, predicts and presents; the human confirms the cause and reason.

The parts involved

Four external systems the app reads or notifies (two more — Keycloak and SVN blame — are flag-gated; see below), its own store, and the app’s own containers.

External systems

Jenkins source of truth — read

The UT pipeline. The app reads, per build: the merged test report (/testReport/api/json), the SVN change sets (the authoritative list of code changes), per-track timing (wfapi/describe), and — for the console-log UT stages — the stage logs (wfapi/log). Anonymous read; token optional.

SVN optional · flag-gated ownership

A second, distinct use of SVN — not the change-set candidates Jenkins already surfaces above. For each failing test, the pipeline maps its source path to the repo and shells out to svn blame --xml (SvnCliBlameClient), taking the modal line author as that test's main developer — this is the dashboard’s Owner, stored on TestIdentity.main_developer. Gated by SVN_BLAME_ENABLED (default off): with it off, no client is built and main_developer stays NULL, so local dev, the offline gate and the public demo never touch SVN. Wired into uta poll/backfill/bootstrap (incremental, per build's failing tests) and the uta reattribute-owners backfill command; the control panel's on-demand ingest intentionally skips it, like email and the LLM.

Oracle ut_ref read-only

Reference DB. Its V_TRACKING view lists data changes (entity, author USRCODE, timestamp). There is no changelog for data — attribution rests on timestamp correlation against the build’s time window. Reached read-only via oracledb thin mode.

LLM provider optional

Anthropic (Claude) or OpenAI, behind a swappable HypothesisProvider. Turns the deterministic signals + similar past cases into a one-sentence, human-readable root-cause hypothesis. With no API key the default Noop provider is a no-op — everything else still works.

SMTP outbound

Two distinct alerts ride this seam. The regression report is sent only when a build worsens (≥1 new failing test vs the baseline) — silence means “no worse than before.” A separate ops alert (subject prefixed UT Analyzer ops —) fires when the poller quarantines/skips a build, or (once, latched) when /health finds the poller stale — operational noise about the pipeline itself, not about test results. Both are disabled when no recipients are configured.

Keycloak (IAM) optional · flag-gated

The company's central identity provider at auth.labsolution.lu, realm labsolution. When AUTH_ENABLED=true (off by default), web acts as a confidential OIDC client (internal-ut-analyzer) doing Authorization Code Flow + PKCE via Authlib: it discovers the realm's endpoints, redirects an unauthenticated browser to Keycloak to sign in, and on /logout also ends the central SSO session (RP-initiated logout). With the flag off the app makes zero Keycloak calls — this is a deployment-time addition, not part of local dev, the public demo, or CI.

The app & its store

PostgreSQL the app’s own store

All build history, test lifecycle, failure episodes, flakiness, and the knowledge base live here. Similarity search uses stock pg_trgm / full-text — no vector database. Reached only via DATABASE_URL (“container now, external later”). Also holds a small operational state — not part of the failure model: setting_overrides (runtime threshold tuning), ingest_jobs (on-demand ingest requests, recovered to ERROR if orphaned by a restart), poller_heartbeats (poller health, now including last_success_at and a latched stale_alerted_at) and build_quarantines (per-build failed-attempt counters and quarantine state).

poller container ingest

Runs uta poll: on a fixed interval, detects new completed builds above the DB high-water mark and drives the ingest + analysis pipeline. The live path — it also sends email and fills LLM hypotheses. Each tick re-reads any control-panel threshold overrides from Postgres (so a tuning change applies with no restart), ends with a retention/pruning pass (old passing results, finished ingest jobs), and stamps a heartbeat row with the tick time, build count and last error. Resilience: transient faults (network, HTTP 5xx/429, DB connection blips) retry in-tick with exponential backoff; a build that keeps failing accrues one attempt per tick and, past the attempt limit, is quarantined — recorded, ops-alerted by email, and skipped so the high-water mark advances past it. A build rotated out of Jenkins retention (404) is quarantined immediately, the explicit form of what used to be a silent skip.

web container dashboard

FastAPI + HTMX/Jinja. Serves the triage dashboard: the three-bucket queue — now with a bookmarkable filter/sort bar (owner, suite, track, predicted cause, triage status, flaky flag; sort by age/name/owner, all via query params) and row-level bulk actions (bulk acknowledge on the New bucket; bulk triage-status/causing-person/reason on Still-failing) plus a one-click “Ack all w/ signature (N)” — shown per New-bucket row only when other unacknowledged failures share its signature — and its attribution twin on the per-test record page (apply cause/reason/Jira ticket/triage status to every open failing test sharing the signature) — build summaries (results table server-side paginated, page size = the ui_row_limit tunable, with an optional failures-only filter), a /builds “Job builds” list (also paginated — every ingested build newest-first — status, start time, duration, test totals, and regression / newly-fixed counts vs the same baseline the build summary uses, with a header banner showing the poller's last and next tick, plus a build-health timeline chart of failed/regression counts over time), per-test records, a flaky leaderboard (each row with a recent-builds sparkline), a global “jump to test” search (navbar box, substring match on canonical name — a unique match redirects straight to the record), and knowledge-base search. Where the human confirms causes and enters reasons. Also serves the /control panel: live-tune the whitelisted thresholds (flaky/KB/triage/ ingest/retention knobs — never secrets or URLs), an AI-suggestion accuracy table (confirmed vs corrected triage verdicts per conclusion field — cause / reason — all-time and over a 30-day window, with a precision figure), trigger an on-demand ingest of a build or range directly against Jenkins/Oracle (back-fill semantics — no email, no LLM; also the recovery path for a quarantined build — a successful re-ingest clears its record), and view poller health (last poll, last successful poll, high-water mark, last error) plus a build quarantine table (failing/quarantined builds with attempts and last error). At startup it also flips any ingest-job row left QUEUED/RUNNING by a prior restart to ERROR, and serves a real /health — DB ping + poller-heartbeat freshness, 503 if the DB is down or no successful poll landed within the configured staleness window (a deployment with no poller, like the public demo, reports poller “never” and stays healthy). Every state-changing request (POST/PUT/PATCH/DELETE) also passes an app-wide CSRF guard — a cross-site browser origin (Sec-Fetch-Site/Origin evidence) gets a 403 before it reaches any route — the same check whether AUTH_ENABLED is on or off. With AUTH_ENABLED=true, a fail-closed middleware gates every route except /health, /login, /auth/callback, /logout and /static/ — an unauthenticated browser GET bounces to /login, a non-GET gets a plain 401 — and the header swaps the free-text actor form for “signed in as <user>” + Logout. Runs behind Traefik in that mode, so uvicorn is started with --proxy-headers to trust X-Forwarded-* (the OIDC callback URL must be the external https one).

migrate + db infra

A one-shot uta migrate brings the schema to head before web/poller start (so they never race). db is the Postgres container in dev; production points DATABASE_URL at an external server.

The monitor human-in-the-loop

Acknowledges new failures, confirms or corrects AI suggestions, enters the causing person / reason / Jira ticket. Every action is stamped with an actor — a self-declared name (Phase 1, default) or the Keycloak-verified preferred_username when AUTH_ENABLED=true (Phase 2) — resolved through one choke point, current_actor, with no data-model change either way.

Deep-link targets read-only links

FishEye (?cs=<revision>) for SVN revisions, Jira (/browse/<TICKET>) for tickets, and ZEPHYR (Kanoah Test Management) for the test case(s) a failing test references (e.g. LX-T4792, parsed from its stack trace) — surfaced as links in the UI, not integrations. Each linked test case is shown alongside its ZEPHYR owner initials (e.g. “(owner kam)”) — the manual test case's owner, parsed zephyr_owner metadata, kept distinct from the dashboard's Owner (the test's main developer, resolved via SVN blame — see above).

System map

How data and notifications flow between the parts. The poller pulls from Jenkins and Oracle, writes to Postgres, prunes old passing results / finished ingest jobs from Postgres at the end of every tick, and fans out to email + LLM; the web app reads Postgres and the human writes back through it. The web app's control panel also has a direct, on-demand path to Jenkins/Oracle for manual ingest — always with back-fill semantics (no email, no LLM). Two producers reach email: the poller for both the regression report and ops alerts (a quarantined/ skipped build), and web's /health check for a latched ops alert when the poller goes stale. A fifth external system, Keycloak, appears only when AUTH_ENABLED=true (pink, dashed — off by default, so it is absent from the map in local dev, the public demo and CI): the browser is redirected to it to sign in, and web ends the central SSO session on logout. A sixth, SVN (blame), similarly appears only when SVN_BLAME_ENABLED=true (amber, dashed): the poller shells out to svn blame per failing test's source file to resolve that test's main developer — the dashboard's Owner — a separate use of SVN from the change-set candidates Jenkins already supplies.

1Ingest loop — poller

The trigger is a scheduled poll (uta poll, default every 300 s). On an empty store it back-fills the last BACKFILL_DEPTH builds oldest-first; then it runs purely incrementally above the DB high-water mark. On-demand uta bootstrap / uta backfill do the same by hand, as does the dashboard's /control panel (a build or range, run in a background thread) — every on-demand path shares the poller's back-fill semantics: no email, no LLM hypothesis, so a re-ingest never re-mails a historical regression. If the re-ingested build is older than the current newest complete build, the lifecycle/classification pass itself is skipped too (see Analysis) — only the build/results/KB signatures are persisted.

  1. Select builds. Everything above the highest ingested build, up to Jenkins’ lastCompletedBuild, minus any quarantined builds. The cursor is the DB itself — restart-safe, no separate state.
  2. Fetch. Per build: build meta, merged test report, SVN change sets, per-track wfapi timing, and (if enabled) the unittest console-log stages’ logs.
  3. Parse. JUnit report → per-(test, track) results; console-log stages parsed into the same shape behind one interface; SVN change sets → code-change candidates; timing → the build window + per-track start/finish.
  4. Correlate data changes. Query Oracle V_TRACKING for changes inside a lookback window before the build (changes precede the nightly build) plus a tolerance margin for clock skew.
  5. Persist. Bulk-insert the build, results, track timings and candidates into Postgres — idempotent on build_number (re-ingest replaces, never duplicates). When SVN_BLAME_ENABLED, also resolves each newly-seen failing test's owner (main developer) via svn blame of its source file, filling TestIdentity.main_developer incrementally.
  6. Record signatures. A normalized failure signature per failing result (the KB recurrence key) — for any build, complete or not.
  7. Prune. Once per tick (not per build), after the builds above are processed: delete passing/skipped results — never failing ones — from builds older than RESULT_RETENTION_DAYS (default 90; 0 disables), and finished ingest jobs older than INGEST_JOB_RETENTION_DAYS (default 30; 0 disables). Also runnable on demand via uta prune. See the Reference invariant below.
Clock discipline is load-bearing. Jenkins timestamps are epoch-millis UTC; Oracle ut_ref timestamps are naive local wall-clock (server is UTC+2), normalized via the named zone Europe/Luxembourg. Data-change attribution rests entirely on these timestamps.

The loop is hardened against faults: a transient error (network, HTTP 5xx/429, a DB connection blip) retries in-tick with exponential backoff; a build that keeps failing accrues one attempt per tick on a build_quarantines row and, past the attempt limit, is quarantined — recorded, ops-alerted by email, and excluded from selection so the high-water mark advances past it. A build rotated out of Jenkins retention (404 on the detail endpoint) is quarantined immediately. A quarantined build's record is cleared by a successful on-demand re-ingest from the control panel.

2Analysis — complete builds only

A build is analysed only when it is complete: all expected tracks reported and each track's stage actually finished running its tests (an aborted/cancelled UT stage keeps the build incomplete, even though its track is present). Incomplete builds are stored and shown but skipped as a baseline, so partial builds never produce phantom regressions. The lifecycle/classification/hypothesis pass also only ever runs for the newest complete build — re-ingesting an older, already-superseded build (e.g. the control panel's quarantine-recovery re-ingest) still persists its build, results and KB signatures, but never replays into current lifecycle state, so history can’t reopen a since-fixed episode or clear an acknowledgement.

Lifecycle & episodes

Each test carries a lifecycle state — FAILING → FIXED, plus REMOVED and a cross-cutting FLAKY flag. A previously-fixed test that fails again opens a new failure episode (regressions are first-class, history is never overwritten). Acknowledgement is an orthogonal attribute, not a state.

Baseline diff

The build is diffed against the most recent complete build: new failures (regressions), newly fixed, still failing, and flaky — each linking to the per-test record. The summary states which build was used as baseline.

Classification

Each new regression gets a deterministic predicted cause — CODE_CHANGE / DATA_CHANGE / INFRASTRUCTURE / UNKNOWN — from the window-filtered code & data candidates, each scored for relevance to that specific test (changed-path overlap with its module/stack frames; changed entity/component named in its error text — coarse tiers, not a probability). When both kinds are present, the tie is broken by score magnitude: the top code candidate's relevance score is compared against the top data candidate's, and the stronger kind wins once it leads by a full relevance tier — otherwise it's UNKNOWN. When the cause is CODE_CHANGE and every candidate SVN commit shares one author (or DATA_CHANGE and every candidate shares one V_TRACKING USRCODE), that person becomes the suggested contact — otherwise it is left blank rather than guessed. Every classification also carries a deterministic confidence in [0, 1] (a “confidence 0.63”-style badge on the per-test record): a flat 0.9 for INFRASTRUCTURE, a flat 0.2 for UNKNOWN, and for CODE_CHANGE/DATA_CHANGE a base of 0.5 plus a term for how decisively the relevance scores favored the winning kind and a term for the strongest knowledge-base provenance already attached to this failure's signature — validated human knowledge feeding back into future predictions (see Learning loop). The evidence (including the tie-break, top matches and confidence inputs) stands on its own — surfaced per episode on the per-test record as a collapsed “Why this prediction” panel — and the human attributes the final cause.

Flakiness

Scored by oscillation, not “no change”: transitions ÷ builds over a 30-day window. Gaps (incomplete/absent) are missing data, never counted as flips. Feeds the flaky leaderboard and the per-test record, both rendering the same windowed pass/fail history as a small SVG sparkline.

3Human triage — web dashboard

The dashboard opens on the daily triage queue — the primary landing surface — organized into the three buckets the monitor thinks in. A filter/sort bar (owner, suite, track, predicted cause, triage status, flaky flag; sort by age/name/owner) sits above the buckets, entirely via bookmarkable query params, so a monitor can save/share a link to “my” slice of the queue.

★ New — unacknowledged

Failing tests with no acknowledgement. Each row's test name carries a muted one-line error snippet (the traceback's closing exception line, falling back to the details field) so triage is possible without opening the record. The action queue: predicted cause + an Acknowledge button, a row checkbox for bulk-acknowledge selected, and — when other unacknowledged failures share the row's signature — a one-click “Ack all w/ signature (N)” showing that blast radius before the click: N is the count of unacknowledged failures (this row included) matching the same exception type + message (own-function stack frames excluded, since a signature key is test-identity-scoped and so never literally shared across tests); the button only appears once N > 1. Driven toward empty.

★ Still failing

Failing and acknowledged — the open caseload, with the same error snippet under the test name, plus age, triage status, causing person, and a one-line cause. Row checkboxes support a bulk “apply to selected” (triage status / causing person / reason) across the caseload. REMOVED mid-failure is flagged distinctly.

★ Recently fixed

Transitioned to FIXED within the last N builds / 7 days — visible and confirmable before it scrolls away.

Each row links into the per-test record: full evidence (candidate code/data changes ranked by relevance to that test, each with its match reason — path overlap or entity mention, most relevant first — rather than one flat build-wide list; see Classification above), the predicted cause's confidence badge with a per-episode, collapsed “Why this prediction” breakdown (infra-error flag, candidate counts, top code/data match with author and score, the tie-break, the confidence-formula inputs), the test's Owner (main developer, resolved via SVN blame), its linked ZEPHYR test case(s) — each shown with its own ZEPHYR owner initials, e.g. “(owner kam)”, distinct metadata from the dashboard Owner — LLM hypothesis, error type, stack trace — clamped past 15 lines behind a “Show full trace” toggle, with a copy-to-clipboard button) plus the editable fields the human owns — causing person, reason for failure, Jira ticket, and triage status. A one-click Confirm sits next to each AI suggestion. When the current failure's signature also afflicts other open failing tests, a second submit button, "apply to all N affected tests with this signature", applies the same cause / reason / Jira ticket / triage status to all of them in one action (an empty Jira field leaves their tickets untouched, so a shared-outage form can't mass-clear unrelated ones); provenance is still derived per test against that test's own AI suggestion — the attribution twin of the New bucket's acknowledge-all-with-this-signature control. Every action is stamped with the acting actor.

4Learning loop — knowledge base

The human’s explanations become a growing, queryable asset — turning one-off knowledge into better future predictions, with no vector database. Two feedback paths close the loop: validated KB provenance raises a future classification's confidence (see Analysis), and every confirm/correct verdict is scored as an AI-suggestion accuracy metric on the control panel.

Failure signatureTest identity + normalized error text (numbers, hex, timestamps, IPs, temp paths, line numbers masked; exception type + top stack frames kept). Stored with a hash for exact lookup. This single function defines what “the same failure” means.
RecurrenceA new failure matching a past signature surfaces “this looks like the failure on <build/date>; previous reason was …; caused by …”.
SimilarityExact hash match + fuzzy pg_trgm trigram / full-text — top-k nearest historical failures, all in stock Postgres.
Provenance tiersEvery conclusion is weighted by how it was reached: AI_UNCONFIRMED (weak) · AI_CONFIRMED · HUMAN_CORRECTED (strongest — the AI was wrong) · HUMAN_ENTERED. Only validated conclusions feed prediction, so the model never reinforces its own guesses. The strongest provenance weight attached to a failure's signature also feeds the classification confidence formula for that signature's future occurrences.
RAG hypothesisThe top-k similar cases are rendered into the LLM prompt, so its one-line hypothesis improves as the KB grows.
AI-suggestion accuracyThe control panel counts AI_CONFIRMED vs HUMAN_CORRECTED triage verdicts per conclusion field (cause / reason), all-time and over a 30-day window, and derives precision = confirmed ÷ (confirmed + corrected). HUMAN_ENTERED conclusions are excluded — with no AI suggestion in play they say nothing about AI quality. This is how Attribution.original_ai_cause/original_ai_reason (retained on correction) get consumed, closing the loop back to the monitor.

5Alerting — email

Because every commit triggers a build, a per-build digest would be constant noise. The tool emails only when a processed build introduces ≥1 new failing test (a regression vs the baseline). Builds with no new failures send nothing — silence means “no worse than before.” The email leads with the new failures (predicted cause + suggested contact each) and includes still-failing / newly-fixed counts as context. When APP_BASE_URL is set, each new failure is followed by a deep link to its per-test record and the message ends with a link to the build's dashboard summary alongside the Jenkins URL; unset (the default), the email stays exactly as before, link-free. A recovery notice (“back to green”) is a separate, optional toggle and gets the same dashboard build link when configured. Everything else stays in the dashboard.

A second, distinct ops alert (subject prefixed UT Analyzer ops —) rides the same SMTP seam but is about the pipeline itself, not test results: the poller sends one when a build is quarantined or skipped (retention 404), and the web app's /health check sends one — latched so repeat probes don't re-mail, and cleared on recovery — when no successful poll has landed within the configured staleness window. These are operational noise, unconditional on test outcomes, so they don't affect the regression-only policy above.

Reference — data model & invariants

The internals a contributor needs: what the store persists to produce the views above, and the facts that silently corrupt the output if they’re gotten wrong. The repo CLAUDE.md is the operating contract; this is the reader-facing companion.

Information model — what is persisted

BuildsBuild #, links, SVN revision(s), overall + per-track start/finish, totals, and the completeness / baseline flag.
Test results per buildIdentity, status, duration, track, failure message, stack trace. These rows are the failure-history feed (counts, last-failed, fail-rate windows) — no separate table. Passing/skipped rows are pruned once their build ages past the retention window (see Retention below); failing rows are kept forever.
Test identity & aliasesCanonical fully-qualified name + optional alias_of pointer so renames/moves keep their history (suggested vs human-confirmed). Also carries the dashboard Owner — main_developer, the modal author from svn blame of the test's source file (identity-level, filled incrementally as failures are ingested, gated by SVN_BLAME_ENABLED) — and, kept as separate ZEPHYR metadata, zephyr_owner (the linked ZEPHYR test case's manual-test owner) and zephyr_test_cases.
Test lifecycleLifecycle state (FAILING / FIXED / REMOVED), the FLAKY flag, reopen_count, and the acknowledgement attribute (flag + acknowledger + timestamp) as a field separate from state.
Failure episodesOne row per fail→fix cycle (first-failure build, fixed-in build — set only when it passed again, never on removal — cause/reason, age), so regressions accumulate instead of overwriting.
SignalsCandidate code changes (SVN update step) and data changes (ut_ref V_TRACKING), infra indicators — each linked to the failure that generated it. (Test ownership is a separate, identity-level attribute — see Test identity & aliases above — not a per-failure signal.)
ClassificationsPredicted cause, a deterministic confidence in [0, 1] (its formula inputs stored in the evidence JSON — relevance-score gap and knowledge-base provenance weight), the LLM hypothesis, and a deterministic suggested contact — the single author/USRCODE behind all candidates for the winning cause, left blank whenever authorship is mixed or unknown (never guessed).
Human inputCausing person, reason text, triage status, Jira ticket, acknowledger; every conclusion carries a provenance tier (AI_UNCONFIRMED / AI_CONFIRMED / HUMAN_CORRECTED / HUMAN_ENTERED), the original AI value when corrected, and who/when.
UsersAn actor string on every human action: a self-declared name in Phase 1 (default test-user, the always-on default), or the Keycloak-verified preferred_username in Phase 2 (AUTH_ENABLED=true) — resolved through one choke point (current_actor), no data-model change either way.
Knowledge baseFailure signatures (normalized error text + its hash for exact lookup, with a pg_trgm GIN index for fuzzy similarity) ↔ entered reasons/causes. All in Postgres, no vector store.

Invariants that silently corrupt if wrong

Clocks. Jenkins timestamps are epoch millis, UTC. Oracle ut_ref CREDATIM/UPDDATIM are naive local wall-clock (server OS is UTC+2) — normalize them via the named zone Europe/Luxembourg (DST-aware), never a fixed +2. Data-change attribution rests entirely on these timestamps.
Test identity is test-levelOne lifecycle per suite/class/method. The track (permanent / permanent_py39) is an attribute, not a separate identity — a result is keyed by (build, test, track) and the track comes from the JUnit suite’s enclosingBlockNames.
Ingest scopeThe primary source is devUTs (nose2) JUnit via /<n>/testReport/api/json (the authoritative ~25k-test surface). The unittest console-log stages are ingested behind the same interface, gated by INGEST_UNITTEST_STAGES with a UNITTEST_SUITES allowlist.
Data-change feedThe Oracle V_TRACKING view as-is (author already resolved as USRCODE). Correlation needs a lookback window — changes precede the build — not just the build’s own start/finish window.
Medical dataLIMS error text, stack traces, and especially MODDATA may contain patient data. Golden fixtures are anonymized/redacted before commit and raw MODDATA is never committed — structure preserved, values redacted.
RetentionNothing else in the store is ever deleted. Only unsigned (signature_id IS NULL) passing/skipped test results from old builds, and finished ingest jobs, are pruned — failing results, builds, episodes, lifecycles, attributions and KB signatures/aggregates are kept forever, so KB occurrence counts are never corrupted. Builds' stored pass/fail/skip totals stay exact forever; only the diff of a build older than the window loses its “newly fixed” entries (accepted). Both windows (RESULT_RETENTION_DAYS, INGEST_JOB_RETENTION_DAYS) are runtime-tunable from the control panel; 0 disables either.

Configuration

Everything is 12-factor: env vars → the typed Settings object (src/uta/config.py), loaded once at process start (also from a local .env in dev). .env.example documents every key with the same defaults and comments as below and is the file to copy when standing up a new environment; this page is the reader-facing explanation grouped by topic. A control panel pill marks the small whitelist (src/uta/control/tunables.py) the /control dashboard page can override live, without a restart — deliberately never secrets, URLs or connection strings, only thresholds/windows; everything else is env-only. A secret pill marks values that must come from a secret store and are never committed.

Deployment

docker compose up brings up four services from one image.

ServiceCommandRole
dbpostgres:16The app’s store (dev only; prod uses an external server via DATABASE_URL).
migrateuta migrateOne-shot: brings the schema to head before the app services start (avoids a migration race).
polleruta pollThe scheduled ingest + analysis loop; the live path for email & LLM.
webuvicorn …The FastAPI/HTMX dashboard on port 8000. GET /health pings the DB and the poller heartbeat — 503 if the DB is down or no successful poll landed within the staleness window; a poller-less deployment (e.g. the public demo) reports poller “never” and stays 200.

Config is 12-factor (env → typed settings). Secrets never committed; .env.example documents every key. Stack: Python 3.12, FastAPI, SQLAlchemy 2.x + Alembic, psycopg (Postgres), oracledb thin (Oracle).

Auth (optional, flag-gated)

With AUTH_ENABLED=true the production deployment runs web behind a reverse proxy (Traefik) so the externally-registered OIDC callback is https; uvicorn is started with --proxy-headers to trust X-Forwarded-*. OIDC_CLIENT_SECRET and SESSION_SECRET come from the deployment secret store, never committed. With the flag at its default (false) — local dev, the public demo, CI — none of this is wired and the app makes zero Keycloak calls.

Public demo

A second, parallel deployment — uta.demo.app:app — hosts a fully synthetic, offline copy of the dashboard for anyone to try, with no connection to Jenkins, Oracle ut_ref, FishEye/Jira, SMTP or an LLM. A fake Jenkins/tracking-feed pair (uta.demo.dataset) is fed through the real ingest + analysis pipeline (uta.demo.seed) into an ephemeral, in-process SQLite store — same code paths, invented data. uta demo serves it locally; uta seed-demo seeds a persistent store instead. render.yaml auto-deploys this image to a free Render web service on every push to main, gated by the same CI check that protects the branch — it is not part of the production system map above. The control panel is read-only in this deployment: its mutating routes (settings save/revert, on-demand ingest) return 403 instead of touching the shared store or opening an outbound connection from the public host, while the health/tunables view and triage actions stay live.
