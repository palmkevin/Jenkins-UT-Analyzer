Help — how to use this dashboard

This tool watches the nightly unit-test run, gathers the evidence for every new failure, and presents it here for a human to triage. It is deliberately assistive: it gathers, correlates, predicts and suggests — a person always confirms the cause and reason. Every action you take (Acknowledge, Confirm, Save) is stamped with the name in the acting as box in the top right, so the triage history says who decided what.

Daily workflow

The dashboard opens on the triage queue — every failing or recently-fixed test, organized into three buckets:

New Failing tests nobody has looked at yet. Each row shows the predicted cause and a one-line error snippet, with an Acknowledge button. Acknowledging just moves it into "Still failing" — it records that a human has seen it, nothing more. When several unacknowledged failures share the same underlying error, an "Ack all w/ signature (N)" button appears so you can clear a whole outage in one click.
Still failing The open caseload: acknowledged and not yet fixed. This is where you set the triage status, causing person and reason — one row at a time on the per-test record, or in bulk across a checkbox selection.
Recently fixed Tests that turned green again within the last {{ recently_fixed_days }} days — visible here for a short window so a fix can be confirmed before it scrolls away.

A filter/sort bar above the buckets (owner, suite, track, predicted cause, triage status, flaky-only) works entirely through the URL, so a filtered view can be bookmarked or shared. Use the search box in the navbar to jump straight to a test by name. Owner, suite and predicted-cause values shown throughout the queue, a run's results and the flaky leaderboard are themselves clickable pivots — click one to jump straight to the triage queue filtered to just that value.

Clicking a test name opens its per-test record: the full failure history, every candidate code/data change ranked by relevance, the AI suggestion (see below), and the editable fields you own — causing person, reason, Jira ticket, triage status. When the same failure signature affects other open tests, a second button lets you apply the same conclusion to all of them at once.

Statuses & badges

Test result (per run)

The raw outcome of one test in one run — always shown with both a colour and a glyph so it reads without colour vision.

✓ PASSEDThe test passed.
✕ FAILEDThe test failed in this run.
○ SKIPPEDThe test did not run (e.g. disabled).
REGRESSIONA failure that is new versus the previous complete run — the thing the daily queue and email digest are built around.
FIXEDPassed again after having failed — shows up in "Recently fixed".

Test lifecycle

The test's ongoing state, independent of any single run.

FAILINGCurrently failing — open in "Still failing" until it passes again.
FIXEDPassing again. A test that fails again later reopens with a fresh failure episode — its history is never overwritten, so repeat regressions are visible.
REMOVEDThe test no longer appears in the report (deleted or renamed). If it was failing when removed, its row is flagged so the episode isn't silently lost.
flakyAn oscillation flag, not a lifecycle state — see Flakiness below. A test can be flaky and failing, or flaky and passing.

Triage status

Set by hand on the "Still failing" caseload — use it to track where each open failure stands.

UNTRIAGEDDefault — acknowledged, nothing decided yet.
INVESTIGATINGSomeone is actively looking into it.
ROOT_CAUSEDThe cause is understood (reason/causing person filled in) but the fix hasn't landed.
RESOLVEDConsidered done — typically once the fix has been confirmed in a later run.

Predicted cause & confidence

A deterministic guess computed for every new regression — no LLM involved (see below for what the LLM adds on top).

CODE_CHANGEAn SVN commit in the lookback window touches this test's module/stack.
DATA_CHANGEAn Oracle ut_ref data change in the lookback window names an entity this test's error mentions.
INFRASTRUCTUREThe error looks environmental (timeout, connection, infra) rather than code/data related.
UNKNOWNNo candidate clearly wins — left for a human to work out.
confidence 0.63A number in [0, 1] next to the predicted cause: a flat 0.9 for infrastructure, 0.2 for unknown; for code/data change it starts at 0.5 and rises with how decisively the evidence favours that cause and with how much validated history (confirmed, not just guessed) this exact failure already has in the knowledge base.

Other badges you'll see

permanentWhich parallel test track (shard) the failure is in — a test can fail in one track or both.
reopened ×NHow many times this test has failed, been fixed, and failed again.

shard-correlatedOn the flaky leaderboard: this test only ever fails in one track, a tell for a shard-specific (rather than genuinely random) flake.

How the LLM contributes

Everything above — the predicted cause, the confidence score, the suggested contact — is computed by plain, deterministic code from the evidence (SVN commits, data changes, error text). The LLM adds exactly one thing on top: a short, human-readable root-cause hypothesis — a single sentence shown as "Hypothesis: …" on the per-test record.

It's given the deterministic signals for that failure plus the top few most similar past failures already recorded in the knowledge base, and asked to explain, in one sentence, what probably went wrong. It never sets a status, never picks a cause, and never acts on its own — it only ever produces that one sentence for a human to read and judge.

If no LLM API key is configured, this is simply blank — everything else in the tool (classification, triage, the knowledge base) works exactly the same either way.

Acting on an AI suggestion

When an episode's prediction carries evidence, a collapsed "Why this prediction" panel sits right next to the Confirm button — expand it to see the top code/data match, how a tie between candidates was broken, and the inputs behind the confidence score, so you can sanity-check the suggestion before you act on it.

On a test record with a predicted cause and/or an LLM hypothesis, you have two ways to respond:

Confirm AI suggestion One click. Takes the suggested contact and the LLM hypothesis exactly as given and saves them as the causing person / reason — cheap, so use it whenever the AI got it right.
Edit the fields yourself Type your own causing person / reason / Jira ticket / triage status and hit Save. If what you type happens to match the AI's suggestion, it's recorded the same as a Confirm; if it differs, your version is saved and the AI's original guess is kept alongside it (shown as "was (AI): …") so nothing is lost.

Every conclusion is tagged with how it was reached — this is what shows in parentheses next to the causing person / reason on the record:

AI_UNCONFIRMEDAn AI suggestion exists but no human has confirmed or corrected it yet — the weakest tier.
AI_CONFIRMEDA human confirmed the AI's suggestion (via Confirm, or by entering the same value by hand).
HUMAN_CORRECTEDA human overrode the AI's suggestion with a different value — the strongest signal, because it tells the tool it was wrong.
HUMAN_ENTEREDThere was no AI suggestion to confirm or correct — entered from scratch.

This matters beyond record-keeping: only confirmed/corrected conclusions (never raw AI guesses) feed back into future predictions — a validated match in the knowledge base raises the confidence score the next time this failure recurs. The same confirm/correct verdicts also drive the AI suggestion accuracy table on the Control panel — confirmed vs. corrected counts and a precision figure, all-time and over the last 30 days, so the team can see whether the AI is actually earning its keep. Entries with no AI suggestion in play (HUMAN_ENTERED) are excluded from that figure, since they say nothing about AI quality.

Knowledge base — reusing past triage

Every failure is reduced to a signature: the test's identity plus its error text with the noisy parts (numbers, timestamps, hex, paths) masked out, so the same failure is recognized again even if the odd detail changes. When a new failure matches a past signature, the record surfaces "this looks like the failure on run …; previous reason was …; caused by …" — someone else's already-confirmed answer, ready to reuse.

The Knowledge base page lets you paste an error message or stack fragment and search across all past failures by similarity, independent of any specific test record. If a search there turns up nothing, it offers to try the same text as a test-name search instead (and the navbar test search does the reverse when it finds no matching test) — the text you typed might have been the other kind of query.

Other pages

Job runsEvery ingested run, newest first — status, duration, pass/fail/skip totals, regressions and newly-fixed counts versus baseline, plus a run-health timeline chart and the poller's last/next tick. A run's failed count links straight to that run's results filtered to failures only.
FlakyTests ranked by oscillation (how often they flip between pass and fail), not just fail count — a solidly-failing test is a regression, not flaky.
Knowledge baseFree-text search over past failures, described above.
ControlLive-tunable thresholds (flaky sensitivity, knowledge-base cutoffs, retention windows, and more — no redeploy needed), the AI suggestion accuracy table, poller health, build quarantine, and on-demand ingest of a specific build or range.
